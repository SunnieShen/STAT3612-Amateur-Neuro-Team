# STAT3612 项目提案执行流程（Proposal Workflow）

## 1. 目标与范围

本阶段目标是完成项目提案所需的第一轮数据研究，并形成可复现、可分工、可汇报的工作流。

- 目标任务：基于 `kaggle-dataset/` 完成数据理解、质量检查、初步特征研究与 baseline 设计。
- 最终交付：一份结构完整的 proposal（问题定义、数据描述、方法路线、实验计划、分工与排期）。
- 本文聚焦：第一步（5人团队如何研究 dataset）并给出具体执行细则。

---

## 2. 项目工作流（从数据到提案）

1. **数据审计（Data Audit）**
   - 统一主键：所有文件按 `case_id` 对齐。
   - 检查 split：`train.json`、`val.json`、`test.json`。
   - 检查字段缺失、异常值、路径可用性、标签分布。

2. **多源数据融合（Data Integration）**
   - 主表：以 JSON 为骨架。
   - 外联：`clinical_information/*.csv`、`original_raw_report/*.csv`、`radiomics_info/*/*.csv`。
   - 输出：统一样本级表（每行一个 `case_id`）。

3. **探索性分析（EDA）**
   - 结构化字段、文本报告、radiomics 分别建探索报告。
   - 结论导向：识别可用特征、潜在泄漏、风险点。

4. **Baseline 方案设计**
   - 先做可跑通 baseline（结构化+radiomics）。
   - 固定验证方式与指标（建议 macro-F1 + accuracy）。

5. **Proposal 撰写与答辩准备**
   - 汇总数据洞察、模型路线、实验计划、风险控制。
   - 明确每位成员后续责任边界。

## 3. 五人分工（角色、任务、交付）

建议采用“角色负责制 + 交叉复核”。

### 成员A：数据质量与规范负责人

- 核心职责
  - 建立数据字典（字段解释、类型、取值范围）。
  - 做数据一致性检查：`case_id` 对齐、重复、缺失、异常值。
  - 输出路径映射规则（JSON 路径与实际目录差异）。
- 必交付
  - `docs/data_dictionary.md`
  - `docs/data_quality_report.md`
  - `scripts/check_data_integrity.py`（可选）

### 成员B：结构化临床特征负责人

- 核心职责
  - 分析 `clinical_information` 各字段分布。
  - 与标签（`Overall_class`）做统计关联。
  - 给出结构化特征处理建议（编码、标准化、缺失策略）。
- 必交付
  - `notebooks/eda_structured.ipynb`
  - `docs/structured_feature_plan.md`

### 成员C：文本报告负责人

- 核心职责
  - 分析 `report` 与 `raw_report` 质量差异。
  - 文本长度、关键词、术语分布与类别关系探索。
  - 提出文本特征路线（TF-IDF 或 embedding）。
- 必交付
  - `notebooks/eda_text.ipynb`
  - `docs/text_feature_plan.md`

### 成员D：Radiomics 负责人

- 核心职责
  - 分析四个序列（`ax_t1`、`ax_t1c`、`ax_t2`、`ax_t2f`）特征分布。
  - 检查 radiomics 异常值、尺度差异、区分度。
  - 提出序列级特征融合方式（拼接/加权/筛选）。
- 必交付
  - `notebooks/eda_radiomics.ipynb`
  - `docs/radiomics_feature_notes.md`

### 成员E：Baseline 与评估负责人

- 核心职责
  - 构建端到端 baseline（训练、验证、预测、提交）。
  - 建立统一评估脚本与结果记录模板。
  - 维护实验日志（参数、版本、分数）。
- 必交付
  - `notebooks/baseline_pipeline.ipynb`
  - `docs/experiment_log.md`
  - `outputs/sample_submission_v1.csv`

## 4. 第一周执行节奏（建议）

### Day 1：规范冻结

- 统一字段命名、目录结构、脚本运行方式。
- 定义“单一真相源”：`case_id` 为唯一 join key。
- 约定评估指标与随机种子。

### Day 2-3：并行研究

- A/B/C/D 并行完成 EDA 与专题报告。
- E 先做可跑通 baseline（不追求最优，只追求稳定复现）。

### Day 4：汇总与决策

- 合并 4 份 EDA 结论，形成 v1 特征清单。
- 确定 proposal 里的方法主线和备选路线。

### Day 5：提案定稿

- 补齐图表、实验计划、风险与备选方案。
- 进行一次组内模拟答辩（10-15 分钟）。

---

## 5. 操作SOP（每个人都要遵守）

1. **先审计再建模**：任何建模前必须完成数据质量检查。
2. **所有拼接必须按 `case_id`**：禁止按行序拼接。
3. **处理 CSV 列名 BOM**：读取后统一清洗列名（去 `\ufeff`）。
4. **路径可用性检查**：加载 `image_path` 前先做 exists 检查与修正映射。
5. **固定验证集**：只用 `train` 训练、`val` 验证，避免数据泄漏。
6. **记录每次实验**：至少记录特征版本、模型参数、分数、结论。

---

## 6. 团队协作机制（避免“各做各的”）

- 每日站会（10 分钟）
  - 昨天完成什么
  - 今天计划做什么
  - 当前阻塞是什么

- 同步节奏
  - 中午：进度短同步（文本）
  - 晚上：成果合并（代码/文档）

- 复核机制
  - 每个 notebook 至少由 1 人复核可复现性
  - 每份报告至少由 1 人复核结论与图表一致性

## 8. 评分导向建议（写入 proposal 可加分）

- 明确说明：类别不平衡问题与应对策略（class weight / resampling / macro 指标）。
- 明确说明：多模态融合策略（结构化 + 文本 + radiomics）的迭代路径。
- 明确说明：可复现实验设计（固定 split、固定 seed、统一日志模板）。
- 明确说明：风险与替代方案（文本效果弱时，先确保结构化+radiomics可交付）。

## 9. 第一阶段完成标准（Definition of Done）

满足以下条件即视为“第一步完成”：

- [ ] 已形成统一数据字典与质量报告。
- [ ] 三类 EDA（结构化/文本/radiomics）全部完成并可复现。
- [ ] baseline 可在本地端到端跑通并输出提交文件。
- [ ] proposal 的方法路线、实验计划、分工排期均已落文档。
- [ ] 团队每人有明确下一步任务与截止日期。
